Tres enfoques de implementación

A

CSS Scroll-driven

Recomendado

Animación nativa del browser. animation-timeline: scroll() expande el clip-path o el width proporcionalmente al scroll. Cero JS, cero dependencias.

✓ Sin dependencias • 0KB JS extra
✓ 60fps garantizado (browser nativo)
✓ Código muy simple
✗ Chrome 115+ / Safari 17+ (~82% global)
✗ Firefox: flag experimental
B

Vanilla JS + rAF

Alternativa ligera

Listener de scroll con requestAnimationFrame. Calcula el progreso (0→1) y aplica clip-path directamente en el estilo del elemento. ~25 líneas.

✓ Sin dependencias • ~1KB JS
✓ Compatible 100% browsers
✓ Patrón consistente con el resto del tema
✗ Más verboso que CSS
✗ Hay que gestionar el cleanup
C

GSAP ScrollTrigger

Más potente / más pesado

La librería de referencia para scroll animations. gsap.fromTo() + scrub: true. Máxima precisión y control, ideal si hay más animaciones complejas en el sitio.

✓ API muy ergonómica
✓ Fácil de extender a otras secciones
✓ Soporte excelente
✗ +37KB gzipped (nueva dependencia)
✗ Overkill si solo es 1 animación
Mi recomendación: Opción A (CSS) como implementación principal + Opción B (vanilla JS) como fallback para Firefox/browsers sin soporte. La combinación da cobertura total sin añadir dependencias. Si más adelante aparecen más animaciones complejas (parallax, stagger, etc.) se puede incorporar GSAP solo entonces.
